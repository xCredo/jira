# Правила модулей

Этот документ закрепляет практики работы с модульным подходом для новых и модернизируемых фич.

## 1) Что считается модулем

- Модуль — это feature с `module.ts` + `tokens.ts`, зарегистрированная через `Module.ensure(container)` в `src/content.ts`.
- Папки модулей называются с суффиксом `-module`, напр.: `person-limits-module`.
- Для старых фич до миграции это соглашение может временно не соблюдаться, но новый код должен идти в `*-module`.
- Приёмный маршрут после миграции: все импорты и интеграции обновляются на новые пути с `-module`.

## 2) `module.ts`

`module.ts` должен:

- Экспортировать один инстанс модуля (`export const ...Module = new ...Module();`).
- Регистрировать все DI токены фичи через `this.lazy(...)`.
- Для `Valtio` моделей использовать `modelEntry(...)`, чтобы получить `model` + `useModel`.
- В React: **`useModel()`** — только чтение state для рендера; **методы модели** — у **`model`** из того же entry (иначе снапшот read-only и команды падают в рантайме).
- Не создавать singleton вручную через `container.register({ token, value: ... })` для тех же сущностей, если есть `lazy + modelEntry`.

## 3) `tokens.ts`

- Содержит все токены данного модуля.
- Для сущностей, попадающих в DI, используется `createModelToken`.
- Токены должны быть покрыты типами:
  - `Token<ModelEntry<T>>` для моделей;
  - `Token<T>` для остальных DI-объектов, если нужен доступ через `inject`.
- Каждый токен обязателен к документированию JSDoc:
  - назначение (`property`, `board-runtime`, `settings-ui`, `pageObject` и т.д.),
  - владелец жизненного цикла,
  - потребители (`BoardPage`, `SettingsPage`, и т.д.).

Пример:

```ts
/**
 * PropertyModel for feature state loaded from Jira board property.
 * Lifecycle: "while board page active".
 * Consumers: BoardPage, SettingsPage.
 */
export const propertyModelToken = createModelToken<PropertyModel>('person-limits-module/propertyModel');
```

## 4) Импорты между модулями

Контракт DI между модулями:

- Один модуль может импортировать `tokens` других модулей.
- Из других модулей **допускаются только**:
  - `tokens` (runtime),
  - `import type` из `types`.
- Из других модулей запрещено импортировать реализации (`модели`, `сервисы`, `PO`, `utils`).

Идея: взаимодействуем по контракту (`token`), а не через внутренности.

## 5) Чеклист перед внедрением нового модуля

- [ ] Есть `module.ts`.
- [ ] Есть `tokens.ts` с типизированными токенами.
- [ ] Папка модуля имеет суффикс `-module`.
- [ ] `content.ts` вызывает `module.ensure(globalContainer)`.
- [ ] `module.ts` регистрирует все DI токены фичи (`property`, `runtime`, `settings`, дополнительные PO/сервисы).
- [ ] Именованные токены имеют JSDoc с владельцем жизненного цикла и потребителями.
- [ ] `token`-импорты проверены на правило межмодульного контракта.

## 6) Дополнение для линтера

Для модулей включается правило `local/module-import-boundary`, которое запрещает импорты из других `src/features/*` в `module.ts`, кроме:

- `tokens` (`.../tokens`),
- `types` через `import type` (`import type ... from .../types`).

### Примечание по внедрению

- В момент миграции старых фичей в `*-module` и после завершения переименований включаем правило для всего проекта.
- До миграции можно держать это правило в CI с временным исключением старых модулей.
